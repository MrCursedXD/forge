# Forge

Describe an app in plain English, get a live, working single-file website back — generated by Groq, saved to a personal gallery backed by Firebase.

## What's in here

```
index.html                  Landing page + login/signup
dashboard.html               The builder: prompt box, live preview, gallery
css/style.css                 Shared styles
js/firebase-init.js          Firebase app/auth/firestore setup (your config is already in here)
js/auth.js                    Login/signup logic
js/dashboard.js               Build/save/load/delete logic
netlify/functions/generate.js Serverless function that calls Groq — this is the ONLY place your API key is used
netlify.toml                   Netlify build/function config
.env.example                   Template for your Groq key (local dev only)
```

Your Groq key is **never** put in any file that ships to the browser. It lives only as a Netlify environment variable, read by `generate.js` on the server side.

## 1. Set up Firebase (one-time)

Your config is already wired into `js/firebase-init.js`, pointing at the `multiuse-c4be6` project. You still need to turn on two things in the [Firebase console](https://console.firebase.google.com/):

1. **Authentication** → Sign-in method → enable **Email/Password**.
2. **Firestore Database** → create a database (production mode is fine) if you don't already have one.

Then add these Firestore security rules (Firestore → Rules). Builds are readable by anyone with the link (that's what makes the "live link" sharable/deployable), but only the owner can create or delete their own builds:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /forge_generated_apps/{docId} {
      allow read: if true;
      allow create: if request.auth != null && request.auth.uid == request.resource.data.uid;
      allow delete: if request.auth != null && request.auth.uid == resource.data.uid;
    }
  }
}
```

The app uses a query that's sorted and filtered together (`where uid == ... orderBy createdAt`), so the first time you load the gallery, Firestore will show a console error with a link to auto-create the needed composite index — just click it once.

## 2. Get a Groq API key

Grab one from [console.groq.com](https://console.groq.com/keys) if you don't already have it handy.

## 3. Deploy

This project ships with **both** a Netlify function (`netlify/functions/generate.js`) and a Vercel function (`api/generate.js`) that do the same thing, so it works on either host without changes. The frontend always calls `/api/generate` — on Netlify that path is redirected to the Netlify function (see `netlify.toml`), on Vercel it's picked up automatically because it's under `/api`.

### Netlify

**Option A — drag and drop:** zip this whole folder and drag it onto [app.netlify.com/drop](https://app.netlify.com/drop).

**Option B — connect a git repo:** push this folder to GitHub/GitLab and "Import an existing project" in Netlify. Build settings are already defined in `netlify.toml`, so you can leave the build command blank.

Either way, **after the first deploy**:

1. Go to **Site settings → Environment variables**.
2. Add `GROQ_API_KEY` with your real key.
3. Trigger a redeploy (Deploys → Trigger deploy → Deploy site) so the function picks up the variable.

### Vercel

1. Push this folder to a git repo and import it in [vercel.com](https://vercel.com), or run `vercel` from inside this folder with the Vercel CLI. No build command is needed — it's a static site plus one file under `/api`.
2. Go to **Project → Settings → Environment Variables**, add `GROQ_API_KEY` with your real key, for all environments.
3. Redeploy (Deployments → ⋯ → Redeploy) so the function picks up the variable.

## 4. Test it

Visit your Netlify URL, create an account, type a prompt like "a pomodoro timer with a circular progress ring," and hit **Build it**. It should stream through a short build log and drop a live preview into the page. Save it to keep it in your gallery, or download it as a standalone `.html` file.

## How "deploying" works here

Every saved build gets a real, shareable URL on your own domain: `yoursite.app/view.html?id=xxx`. That page pulls the build's HTML straight from Firestore and renders it full-page — so it's genuinely live and linkable, not just a preview inside the dashboard. This is what the "Live link" box shows after you save a build.

This is different from spinning up a brand-new, separate Vercel/Netlify project per build (the way you'd manually deploy a new site). That's possible but heavier: it needs a Vercel or Netlify API token stored as a secret, and your server calling their Deployments API to create a new project per build. If you want that later, it's a clean next step on top of what's here — just ask.

## Notes & next steps

- Generation uses `openai/gpt-oss-120b` on Groq at high reasoning effort, which is the strongest model currently available there for this kind of task. If output quality still feels off, the system prompt in `api/generate.js` (and the mirrored `netlify/functions/generate.js`) is the place to tune it — it's written in plain English, not code.
- The generator asks the model for a single self-contained HTML file (inline CSS/JS), so anything it builds is instantly previewable in an iframe, portable as one file, and servable from `view.html`.
- The current setup doesn't verify the Firebase auth token on the server before calling Groq — anyone who can reach your deployed URL and knows the function path could call it directly. For a public/production launch, consider adding `firebase-admin` to verify the user's ID token before spending Groq credits.
- Local development: copy `.env.example` to `.env`, fill in your key, and run `netlify dev` (requires the Netlify CLI: `npm install -g netlify-cli`) or `vercel dev` (requires the Vercel CLI: `npm install -g vercel`).
